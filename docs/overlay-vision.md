## Problem Statement

Knowledge workers at startups pay for 3–5 AI subscriptions. None of them share context. Every session starts from scratch. The workaround is manual copy-paste, re-explaining the same company context, and hoping the AI remembered anything.

Meanwhile, OpenAI and Google are shipping workspace agents — but they're closed, they own your data, and they can't be customized. As software production gets cheaper, teams want tools they can fork, extend, and own.

## Target User

**Who:** Non-technical knowledge worker at a startup — marketing, ops, product. Heavy AI user. Frustrated that AI doesn't know their company. Paying for ChatGPT and Claude and Notion AI and none of them talk to each other.

**What they want:** One AI workspace that knows their context, runs their recurring tasks, and connects to their tools. Not just a chat window — a workspace.

**Why Overlay:** It's open source. They can self-host it. They can customize it. And it will have agents they can configure like employees and automations they can describe in plain language.

## Positioning

**One-liner:** The all-in-one open source AI workspace.

**Tagline (internal direction):** Hire your first AI team.

**Why open source wins:**
- Data sovereignty: companies don't trust OpenAI/Google with their internal knowledge
- Composability: MCP marketplace + community agents = ecosystem velocity closed tools can't match
- Customizability: as AI makes custom software cheap, "forkable" beats "locked in"
- Trust: users own their agents, their data, their workspace

## Competitive Context

OpenAI launched Workspace Agents (April 2026) — shared always-on Codex agents, 60+ enterprise integrations, schedulable. Google shipped Workspace Studio — no-code agent builder across Gmail/Docs/Sheets.

**They can't build:**
- A community-driven agent + automation registry
- Self-hostable, privacy-first, forkable workspace
- Graphical GUI on top of ANY MCP server (not just their own apps)

## Architecture: Three Layers

```
LAYER 1 (exists today)
  Chat + Projects + Knowledge + Notebooks + Skills + MCP + Memory
  → The workspace foundation

LAYER 2 (next 3 months)
  Agents (AI employees) + Automations (scheduled workflows)
  → Makes the workspace proactive, not just reactive

LAYER 3 (6–12 months)
  Graphical MCP App Marketplace
  → Turns Overlay from a workspace into a platform
```

## Recommended Approach: Platform-First

Build agents and automations as first-class architecture. Don't wait for more "wedge" validation — the foundation is solid enough to re-launch with differentiated capabilities.

### Agents (AI Employees)

An agent is a named, persistent AI worker with a role, a set of tools, memory, and a task queue.

**Conceptual model:** Hiring an employee, not configuring a bot.
- Give them a name and a role ("Content Writer", "Customer Support Bot", "Research Assistant")
- Assign them tools (MCP servers they can call)
- Give them skills (instructions that define how they work)
- Set their memory scope (what project/knowledge base they can access)
- Set their approval policy (what actions need human sign-off)

**UX: Agent "Profile"**
```
┌─────────────────────────────────────────────────────────────┐
│ ◉ Content Writer                          [Active] [Settings]│
│   Role: Draft content based on brand voice                  │
│   Tools: Google Docs, Notion, Web Search                    │
│   Memory: Marketing Project                                 │
│   Approval required: Publish, Send                          │
├─────────────────────────────────────────────────────────────┤
│ RECENT ACTIVITY                                             │
│ ✓ 2h ago  Drafted 3 LinkedIn posts from brief               │
│ ✓ 5h ago  Researched competitor blog topics                 │
│ ⏳ Now     Waiting for approval: "Post this draft?" [Y] [N] │
└─────────────────────────────────────────────────────────────┘
```

**UX: Agents sidebar (left nav)**
```
My Agents
├── + Hire New Agent
├── ● Content Writer         (active)
├── ● Customer Support Bot   (2 pending)
└── ○ Research Assistant     (paused)
```

**"Hire" flow:**
1. Browse template gallery or start from scratch
2. Name them, pick a role
3. Assign tools (drag MCP servers in)
4. Write or pick a skill (instructions)
5. Set approval gates
6. "Hire" → agent is live, waiting for tasks

**Task routing:**
- User @mentions agent in chat → agent picks up the task
- Agent produces output → goes into task queue
- High-stakes actions (send email, post, modify data) → approval gate appears in workspace

**Approval UX:** Ambient notification in workspace: "Content Writer wants to post this draft to LinkedIn — [Preview] [Approve] [Edit] [Reject]"

### Automations (Scheduled AI Workflows)

An automation is a trigger + a sequence of AI steps + tool calls. Described in plain language. The AI figures out the steps.

**Conceptual model:** A script you describe, not a flow you draw.

**UX: Create automation (plain language)**
```
┌─────────────────────────────────────────────────────────────┐
│ What should this automation do?                             │
│                                                             │
│ "Every Monday morning, pull my GitHub PRs from the week,   │
│  summarize what I shipped, and email me a standup brief."  │
│                                                             │
│                              [Generate Steps →]            │
└─────────────────────────────────────────────────────────────┘

→ AI generates:
  Trigger: Every Monday at 8am
  Step 1: Fetch GitHub PRs merged in last 7 days
  Step 2: Summarize each PR in 1-2 sentences
  Step 3: Format as standup brief
  Step 4: Send email to divyansh@layernorm.co

  [Edit] [Activate]
```

**UX: Automation list**
```
My Automations
├── + New Automation
├── ● Weekly Standup         Mon 8am   Last: ✓ 3 days ago
├── ● Competitor Monitor     Daily     Last: ✓ 2h ago
├── ● Lead Enricher          On event  Last: ✓ 12m ago
└── ○ Newsletter Draft       Weekly    Paused
```

**History view:** Every run is a full chat transcript + tool call log. Click any run to see exactly what the AI did and why.

**Error handling:** "Automation failed — Step 2 returned an error. [See transcript] [Retry] [Edit]"

**Trigger types:**
- Schedule (cron)
- Webhook (external event)
- Manual ("Run now")
- Event-based (new row in Airtable, new email from X, new GitHub issue)

### Graphical MCP App Marketplace (Layer 3)

MCP servers today are text-only. The MCP Apps spec (Jan 2026) allows interactive HTML interfaces that render inside AI clients.

Overlay's advantage: render these GUIs natively in the workspace sidebar, not just in chat.

**Phase 1 (now):** MCP server directory — install, configure, assign to agents
**Phase 2 (6 months):** Community-built agents and automation templates for specific MCP servers
**Phase 3 (12 months):** Graphical MCP Apps — install apps that render full UI panels in the Overlay sidebar. CRM dashboard, calendar view, analytics panel — all AI-powered, all from the community.

```
MCP Marketplace
├── Featured Apps
│   ├── [GitHub] Graphical PR dashboard  ⬇ Install
│   ├── [Linear] Sprint board + AI triage ⬇ Install
│   └── [Airtable] Smart table with AI    ⬇ Install
├── Agent Templates
│   ├── Customer Support Bot             ⬇ Use template
│   ├── Content Calendar Manager         ⬇ Use template
│   └── Sales Research Assistant         ⬇ Use template
└── Automation Templates
    ├── Weekly standup brief             ⬇ Use template
    └── Lead enrichment pipeline        ⬇ Use template
```

## The Assignment

Before building agents or automations, do this first:

**Re-launch to 10 people.** Not 168. Ten. Find 10 non-technical knowledge workers at startups. Give them Overlay today (as-is). Watch them use it — don't help. Write down what they try to do that the product can't do.

What they try to do but can't — that's your agents+automations feature list. Don't build from imagination, build from observed frustration.

The agents and automations design above is directionally right. But the specific triggers, the specific tools, the specific approval UX — you'll get those details wrong without watching 10 real users try to use it.

**Then:** build agents + automations for the top 3 observed frustrations. Re-launch to 100.

## What I Noticed

You already have almost everything you need:
- `Skills` = the instructions layer for agents
- `convex/crons.ts` = the scheduler for automations
- `Daytona` = the execution environment for agent tasks
- `McpServersView` = the tool layer for agents
- `MemoriesView` = the memory layer for agents

You're not building agents from scratch. You're composing them from primitives you already have.

The missing pieces are:
1. An agent runtime (persistent task queue + approval gate)
2. An automation runner (trigger → step execution → logging)
3. UX for configuring agents and automations in human terms ("hire" and "describe in plain language")
